# 飞行性能分析

用户分析请求：

```text
$ARGUMENTS
```

使用本技能把用户的飞行性能分析目标转化为可确认、可计算、可追溯、可保存的分析流程。默认使用中文工作，并在回答中呈现可审计的“分析思路摘要”，而不是只给最终数值。

## 工作准则

1. 先识别目标，再计算。把用户话语归类为：输入数据整理、剖面定义、重量状态评估、单项性能计算、完整任务性能分析、剖面优化、飞行包线绘制、结果归档或报告生成。
2. 每轮关键输出前给出简短的“分析思路摘要”：说明当前目标、已知输入、缺失项、采用的方法、下一步确认点。不要输出逐字内部推理链；要输出工程上可复核的判断依据和假设。
3. 关键输入、模型选择、中间状态和最终结果必须留有确认记录。若用户已经明确授权继续，可把确认记录写为“按用户已给条件执行”，但不能省略输入状态和假设标注。
4. 所有数值都标注来源：用户输入、本地资料检索、计算得到、工程假设、或外部文件。不要把假设伪装成已验证数据。
5. 优先复用项目能力：本地 `RAG-data` 用于航空资料依据，`aircraft-design-rag` 用于检索，`aircraft-conceptual-design` 的约束图脚本可用于方案界限线图。
6. 缺少气动、动力、重量或环境数据时，先给出最小可运行数据清单和建议取值范围，再请用户确认；只在用户允许时使用工程假设继续。
7. 涉及武器、打击、规避或伤害优化时，只讨论飞机总体性能、载荷包线、重量、布置和飞行安全约束，不提供战术使用或杀伤优化细节。

## 快速流程

1. **意图识别**：提取分析目标、飞机类型、构型状态、任务阶段、关注指标、交付物和精度等级。
2. **输入台账**：整理气动数据、动力数据、重量状态、飞行剖面、大气/机场条件、限制条件和数据来源。详细字段见 `references/data-contract.md`。
3. **确认节点 A**：向用户确认输入台账、单位体系、缺失项和允许使用的假设。
4. **流程编排**：按目标选择计算模块。完整任务通常按“数据校核 -> 剖面建模 -> 重量递推 -> 性能计算 -> 约束/包线 -> 优化 -> 结果归档”执行。
5. **确认节点 B**：确认计算模型、公式来源、剖面段、重量状态、优化目标和约束。
6. **执行计算**：使用用户数据、项目脚本、必要的临时代码或表格计算。每个关键步骤记录输入、输出、单位和状态。
7. **确认节点 C**：展示关键中间结果，例如阻力极曲线适用范围、推力/功率修正、燃油或能量递推、限制边界和初步性能指标。
8. **输出与保存**：用 `scripts/write_performance_bundle.py` 归档 JSON、CSV、Markdown 和 SVG 图表；若需要方案界限线图，调用总体设计技能中的 `plot_constraint_boundary.py`。
9. **最终回答**：给出分析思路摘要、确认记录、关键结果、图表/文件路径、风险和下一步建议。

## 人机确认点

在以下节点主动请求或记录确认：

- **目标确认**：分析对象、目标指标、精度层级、输出格式。
- **特性数据确认**：气动极曲线、升力/阻力/俯仰特性、发动机/电机/螺旋桨特性、安装损失、推力或功率修正。
- **剖面确认**：起飞、爬升、巡航、待机、盘旋、下降、着陆等航段的高度、速度、时间、距离、推力设置和构型。
- **重量确认**：起飞重量、空机重量、有效载荷、燃油/电池、任务段重量递推、重心或重量包线。
- **计算确认**：大气模型、单位体系、性能公式、插值/外推方法、约束边界和适用范围。
- **优化确认**：优化目标、变量、约束、不可违反条件、终止准则和可接受折中。
- **结果确认**：关键性能结果、异常值、图表解释、保存位置和是否生成报告。

确认话术要短而具体。例如：“我将用用户给定的 `CLmax_landing=2.4` 和 ISA 海平面条件计算着陆翼载上限；若未确认，我会把它标为工程假设。”

## 数据与模型管理

输入数据按五类管理：

- **气动特性**：`CL-alpha`、`CD0`、`K`、极曲线、`CLmax`、构型修正、压缩性/雷诺数/结冰/外挂修正。
- **动力特性**：海平面静推力或功率、推力高度/速度修正、SFC/耗油率、油门设置、安装损失、电推进效率和电池能量密度。
- **重量状态**：起飞、爬升初始、巡航初始、着陆、任务载荷、燃油/能量、备用油、重心范围。
- **飞行剖面**：航段名称、模式、速度/Mach、高度、时间/距离、爬升率、转弯/过载、构型、推力设置。
- **环境与约束**：ISA 偏差、机场高度、跑道长度、风、温度、最大动压、结构过载、失速裕度、升限、噪声或运行限制。

同一参数出现多个来源时，优先级为：用户明确指定并确认 > 项目数据文件 > 本地 RAG 证据 > 计算反推 > 工程假设。单位不一致时先统一再计算，并在输出中保留原始单位。

## 计算模块

按用户目标选择最小必要模块：

- **起飞/着陆**：跑道距离、越障、失速速度、翼载上限、推重比或功率载荷要求。
- **爬升/升限**：爬升率、爬升梯度、剩余推力/功率、实用升限、绝对升限。
- **巡航/航程/航时**：巡航速度、高度、升阻比、耗油率或能耗、航程、航时、备份燃油/能量。
- **机动/盘旋/加速**：稳态盘旋、瞬时盘旋、过载、转弯半径、加速时间、能量机动指标。
- **飞行包线**：`V-n`、高度-速度、Mach-高度、失速边界、动压边界、推力/功率边界、结构边界。
- **剖面优化**：以燃油/能量最小、航程最大、时间最短、爬升效率最高或约束满足为目标，明确变量和边界后再运行。

完整剧本、数据质量检查和常见图表选择见 `references/analysis-playbooks.md`。

## 本地资料与脚本

需要依据资料时，先使用项目 RAG 检索：

```bash
python "${CLAUDE_PROJECT_DIR}/.clawd/skills/aircraft-design-rag/scripts/search_rag.py" --data-dir "${CLAUDE_PROJECT_DIR}/RAG-data" --query "<检索词>" --top-docs 24 --max-hits-per-file 2 --top-k 8
```

建议检索词围绕当前模块拆短：`飞机性能 航程 航时`、`飞行包线 失速 过载 动压`、`起飞 着陆 翼载 CLmax`、`爬升率 剩余推力 升限`、`发动机 推力修正 耗油率`、`飞机设计手册 飞行性能`。

归档结果时，先组织一个性能结果 bundle JSON，再运行：

```bash
python "${CLAUDE_SKILL_DIR}/scripts/write_performance_bundle.py" --input "<bundle.json>" --output "${CLAUDE_PROJECT_DIR}/outputs/performance-analysis/<case-id>"
```

若需要方案界限线图，可复用：

```bash
python "${CLAUDE_PROJECT_DIR}/.clawd/skills/aircraft-conceptual-design/scripts/plot_constraint_boundary.py" --input "<boundary.json>" --output "<boundary.svg>" --csv "<boundary.csv>"
```

## 输出格式

最终回答按任务规模裁剪，但完整分析应包含：

- `分析思路摘要`：目标、输入状态、方法、确认点和主要假设。
- `输入台账`：气动、动力、重量、剖面、环境、约束，标注来源和确认状态。
- `流程编排`：已执行步骤、跳过步骤及原因、关键中间输入输出。
- `性能结果`：指标表、单位、适用条件、裕度、敏感参数。
- `优化/包线`：目标函数、约束、推荐剖面或包线边界、图表解释。
- `文件产物`：JSON、CSV、Markdown、SVG 或其他图表的绝对路径。
- `风险与下一步`：缺失数据、外推风险、需要 CFD/风洞/飞试/MDAO 或更高保真模型确认的内容。

保存文件时使用 `outputs/performance-analysis/<case-id>/`，并至少保留：`case.json`、`summary.md`、`features/`、`profiles/`、`results/`、`figures/`、`records/`。
