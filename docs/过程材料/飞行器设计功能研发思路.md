# 飞行器设计功能研发思路

## 目标定位

本功能的目标不是做一个普通聊天入口，而是把 Clawd Code 改造成“可调用本地资料、可执行工具、可生成工程设计材料”的飞行器总体设计辅助系统。

目标用户包括：

- 做飞行器总体设计课程或项目的学生。
- 需要快速形成无人机/固定翼概念方案的研发人员。
- 需要将本地资料库、公式计算和设计报告串起来的工程使用者。

## 研发主线

### 1. 能力入口领域化

网页端只暴露“飞行器设计”能力，不直接显示内部技能名和检索参数。这样可以避免用户理解工程实现细节，把交互焦点放在设计任务上。

### 2. 本地资料增强

本地 `RAG-data` 是飞行器设计问答的知识来源。系统先检索资料，再结合工程假设做方案推导。资料增强的核心价值是：

- 给关键判断提供来源。
- 减少纯模型记忆带来的不稳定。
- 让结果能回溯到本地文献或手册片段。

### 3. 工具闭环

飞行器设计不是单轮回答即可完成，通常需要：

- 检索资料。
- 提取参数。
- 估算重量、翼载、推重比。
- 生成约束边界数据。
- 绘制 SVG/CSV 输出。
- 最终合成设计说明。

因此运行时需要控制工具调用回合，防止长时间停留在工具探索阶段。当前改动已加入技能运行回合限制和最终合成 fallback。

### 4. 前端工作台化

网页端承担“可观察”和“可交付”的角色：

- 展示用户输入与助手回答。
- 展示资料证据和工具活动。
- 支持公式渲染。
- 固定模型配置，减少使用前的参数选择。
- 用中文界面贴合飞行器设计使用场景。

## 技术选择

| 技术点 | 当前选择 | 原因 |
|--------|----------|------|
| 模型 | `deepseek-v4-pro` | 用户指定，适合中文工程问答和较长输出。 |
| 网页服务 | Python 标准库 HTTP 服务 | 当前项目轻量，不引入额外前端构建链。 |
| 前端渲染 | 内嵌 HTML/CSS/JS | 便于单文件启动和本地部署。 |
| 资料检索 | 本地 Markdown 语料 + 检索脚本 | 便于复现，不依赖远端知识库服务。 |
| 公式显示 | 本地轻量 LaTeX 渲染 | 避免 CDN 依赖，满足常见工程公式展示。 |
| 约束图 | Python 脚本生成 CSV/SVG | 便于后续报告、论文或演示材料复用。 |

## 后续研发建议

1. 将网页端过程事件写入会话持久化文件。
2. 把 `.clawd/generated` 中的示范数据整理成标准 benchmark。
3. 增加约束分析输入表单，减少自然语言解析误差。
4. 为飞行器设计任务增加结构化输出 schema。
5. 将约束界限线图生成集成到网页端下载入口。
6. 支持资料证据点击跳转到本地 Markdown 原文片段。
7. 增加“设计假设表”和“风险清单”的自动生成。

## 风险与边界

- 本地资料质量决定回答可追溯性，需要持续清理重复、低质量或无关资料。
- 概念设计结果不能直接替代详细设计、CFD、风洞或结构校核。
- 公式渲染目前覆盖常用工程表达，不等价于完整 LaTeX 引擎。
- 长任务仍可能受模型响应速度、工具调用次数和上下文长度影响。
