# 网页端过程可见化设计说明

## 背景

用户希望网页端的风格更接近 Codex 中的对话体验：界面安静、白底、信息密度适中，并且能够看到问题执行过程。这里的“过程可见”不是展示模型隐藏推理，而是展示可审计的执行轨迹，例如资料检索、工具调用、权限处理、耗时和完成状态。

## 设计原则

1. 不展示模型隐藏思维链。
2. 展示可验证的过程事件。
3. 让默认界面足够干净，细节通过折叠展开。
4. 面向飞行器设计场景命名，不把内部工程术语直接暴露给用户。
5. 保留已有能力：流式输出、复制按钮、LaTeX 渲染、资料证据展示。

## 交互方案

每条助手回答上方增加一行过程状态：

```text
处理中 · 3s · 资料检索 · 已附加本地资料证据 · 命中=5 >
已处理 · 18s · 过程记录 2 >
```

折叠展开后展示“过程记录”，记录内容来自现有工具事件数组：

- `rag_retrieval`：资料检索事件，展示命中数量、索引状态、证据片段。
- `tool_use`：工具调用事件，展示工具名称和调用摘要。
- `tool_result`：工具结果事件，展示结果摘要和必要预览。
- `permission`：权限确认事件，展示是否自动批准。
- `request`：前端请求失败或中止时的状态记录。

## 视觉调整

原网页端使用较重的渐变背景、卡片阴影和大圆角。新的方向更接近工程工作台：

- 页面背景改为浅灰白。
- 主对话区使用白底。
- 消息宽度居中，减少漂浮卡片感。
- 助手回答不再使用厚重气泡，而是接近文档式文本。
- 用户消息保留轻量气泡，用浅蓝灰区分。
- 过程行使用上下细分割线，保持可扫描但不抢占正文。

## 与飞行器设计场景的结合

前端只向用户呈现“飞行器设计”能力，不显示内部 `aircraft-design-rag` 名称，不开放资料检索参数。这样可以让用户关注设计任务本身：

- 任务需求。
- 总体参数。
- 约束边界。
- 动力与布局方案。
- 本地资料证据。
- 下一步计算或校核。

## 当前实现位置

主要改动集中在：

```text
src/web/app.py
```

关键前端函数包括：

- `createMessage`：创建用户、助手和系统消息。
- `updateProcessPanel`：更新“处理中/已处理”过程行。
- `createEventList`：渲染过程记录列表。
- `createEvidencePanel`：渲染本地资料证据。
- `sendPrompt`：在流式响应过程中收集工具事件并刷新过程行。

## 后续方向

- 给过程记录增加更明确的时间戳。
- 将不同事件类型配上轻量图标。
- 在回答完成后显示 token 使用量。
- 将过程事件持久化到会话历史中，历史会话恢复后也能保留完整过程记录。
- 加入前端端到端测试，覆盖发送、流式完成、展开过程记录和公式渲染。
