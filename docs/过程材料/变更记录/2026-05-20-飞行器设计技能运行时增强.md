## 改动背景

飞行器总体设计任务往往需要多轮工具调用：检索资料、读取片段、计算约束、生成图表、合成报告。早期运行时容易出现两类问题：

- DeepSeek/OpenAI-compatible 模型返回的工具调用格式与既有解析不完全一致。
- 长任务可能持续调用工具，最终回答不稳定或返回过慢。

## 研发思路

核心思路是让“技能运行”从一次普通聊天升级为受控工作流：

1. 识别 DeepSeek DSML 工具调用格式。
2. 支持 OpenAI-compatible gateway 下的工具调用与流式行为。
3. 给技能执行设置工具回合上限。
4. 达到上限时进入最终合成，避免只留下工具调用痕迹。
5. 在 REPL 中改进技能运行的流式显示。

## 关键设计

### 工具调用解析

不同 Provider 对工具调用的返回结构不完全一致。运行时需要将模型响应统一转换为内部工具调用事件，再交给 Agent Loop 执行。

### 回合上限

飞行器设计技能默认限制工具回合，常规任务不会无限检索。达到上限后，系统转入无工具最终回答，保证用户能拿到可阅读结果。

### 最终合成 fallback

当工具调用已经提供足够证据，但模型仍试图继续调用工具时，fallback 机制会促使模型基于已有资料总结最终方案。
